## 一、这个系统是干什么的？

### 一句话概括

**这是一个"网上社团管理平台"，类似于一个小型的"社团版微信小程序"。**

想象一下大学里的场景：

- 你是个学生，想加入吉他社，但不知道怎么报名
- 社长要统计谁交了社费、谁没交、谁参加了活动，一堆表格很头疼
- 学校管社团的老师要审核每个社团是否合规，看看整体数据

**这个系统就是为了这些问题而开发的。**

### 具体功能

| 功能 | 通俗解释 | 解决什么问题 |
|------|----------|--------------|
| 用户注册登录 | 学生用学号注册账号，输入密码登录 | 统一身份管理，防止陌生人乱操作 |
| 社团创建与管理 | 学生可以申请成立新社团，社长可以管理社团信息 | 社团成立流程规范化，告别纸质申请 |
| 活动发布与报名 | 社长发布活动，学生在线报名参加 | 信息透明，避免"听说有活动但不知道详情" |
| 入团申请与审核 | 想加入社团就提交申请，社长在线审批 | 告别"加个群就算入团"的随意方式 |
| 缴费通知与记录 | 社长发缴费通知，学生在线确认 | 不用挨个催社费，记录清晰可查 |
| 系统通知 | 管理员或社长发布通知，学生能看到 | 避免通知被群消息淹没 |
| 数据看板 | 用图表展示社团数量、活动参与度等统计 | 一目了然地了解运营情况 |

## 二、系统是怎么工作的？

### 整体架构（像盖房子一样理解）

这个系统采用**前后端分离**架构，意思是：

- **前端** = 房子的"装修"（用户看到的界面）
- **后端** = 房子的"地基和结构"（处理数据和业务逻辑）
- **数据库** = 房子的"储藏室"（存放所有数据）

```
用户（浏览器）  ←→  前端（Vue页面）  ←→  后端（Java服务）  ←→  数据库（MySQL）
```

**类比说明：**
- 你打开浏览器看到的页面 = 餐厅的菜单（前端）
- 厨房里做菜的厨师 = 处理请求的后端
- 食材仓库 = 存储数据的数据库
- 服务员传递点菜单 = 前后端之间的API通信

### 用户角色

系统有三种角色，权限不同：

| 角色 | 是谁 | 能做什么 |
|------|------|----------|
| **普通用户** | 注册的学生 | 浏览社团、申请入团、报名活动、查看通知、交社费 |
| **社长** | 社团负责人 | 管理社团、发布活动、审核报名、发缴费通知 |
| **管理员** | 学校老师/系统管理员 | 审核社团、管理所有用户、发布系统通知、查看日志 |

## 三、项目文件夹详细说明

### 总体结构

```
club_manager/           ← 项目根目录，所有文件都在这里
├── frontend/           ← 前端代码（用户看到的网页界面）
├── backend/            ← 后端代码（服务器端的业务逻辑）
├── sql/                ← 数据库脚本（建表语句）
├── 论文材料/            ← 论文相关文档
└── README.md           ← 项目说明文件
```

### 前端文件夹详解（frontend/）

前端就是用户在浏览器里看到的页面，用 Vue 3 框架开发。

```
frontend/
├── src/               ← 源代码目录，所有前端代码都在这里
│   ├── api/           ← "传话筒"目录 - 负责和后端服务器通信
│   ├── components/    ← "积木"目录 - 可复用的小组件
│   ├── composables/   ← "工具箱"目录 - 可复用的逻辑代码
│   ├── layout/        ← "框架"目录 - 页面整体布局
│   ├── router/        ← "导航"目录 - 控制页面跳转
│   ├── stores/        ← "记忆"目录 - 存储全局数据
│   ├── styles/        ← "化妆"目录 - 页面样式美化
│   ├── utils/         ← "百宝箱"目录 - 工具函数
│   ├── views/         ← "房间"目录 - 各个功能页面
│   ├── App.vue        ← 根组件（整个应用的入口）
│   └── main.js        ← 启动文件（告诉浏览器开始运行）
├── package.json       ← 依赖清单（列出了用到的所有工具库）
└── vite.config.js     ← 构建配置（告诉工具怎么打包代码）
```

#### api/ 文件夹 - "传话筒"

**作用：** 这里的每个文件负责和后端的某个功能模块通信。当前端需要获取数据或提交操作时，通过这些文件发送请求。

**通俗理解：** 就像你去餐厅点菜，api/ 就是服务员，负责把你的需求告诉厨房（后端），然后把做好的菜（数据）端回来。

| 文件 | 负责什么 |
|------|----------|
| auth.js | 登录、注册相关的请求 |
| club.js | 社团相关的请求（创建、查询、修改社团） |
| activity.js | 活动相关的请求（发布活动、报名活动） |
| apply.js | 入团申请相关的请求 |
| user.js | 用户相关的请求（修改个人信息等） |
| notification.js | 通知相关的请求 |
| payment.js | 缴费相关的请求 |
| dashboard.js | 数据看板相关的请求 |
| system-log.js | 系统日志相关的请求 |

#### components/ 文件夹 - "积木库"

**作用：** 存放可以重复使用的UI组件。就像乐高积木，可以在不同页面中重复使用。

**当前组件：**
- `ThemeSwitch.vue` - 日夜模式切换按钮，点击可以在浅色和深色主题之间切换

#### composables/ 文件夹 - "工具箱"

**作用：** 存放可以复用的逻辑代码，是Vue 3的特色功能。

**通俗理解：** 就像厨房里的万能工具——削皮器、开瓶器，做好一个可以反复用，不用每次做饭都重新做一个。

| 文件 | 作用 |
|------|------|
| useDialog.js | 弹窗（对话框）的通用逻辑 |
| usePagination.js | 分页功能的通用逻辑 |
| useSearch.js | 搜索功能的通用逻辑 |

#### layout/ 文件夹 - "房屋框架"

**作用：** 定义页面的整体布局结构，像房子的框架一样，决定了页面的整体外观。

**通俗理解：** 每个网页都有固定的"外框"——上面有导航栏、左边有菜单栏、右边是内容区。这个文件夹就负责这些"外框"。

| 文件 | 作用 |
|------|------|
| index.vue | 整体布局的主组件 |
| Header.vue | 页面顶部栏（显示logo、用户信息等） |
| Sidebar.vue | 左侧菜单栏（功能导航） |
| NotificationDialog.vue | 通知弹窗 |
| PasswordDialog.vue | 修改密码弹窗 |

#### router/ 文件夹 - "导航图"

**作用：** 控制页面跳转，决定用户点击某个链接时显示哪个页面。

**通俗理解：** 就像地图导航，告诉系统："当用户访问 /club 时，显示社团列表页面；访问 /activity 时，显示活动页面"。

#### stores/ 文件夹 - "记忆库"

**作用：** 存储全局共享数据，比如当前登录的用户信息、主题设置等。

**通俗理解：** 就像你口袋里的钱包，不管走到哪都能取出来用，不用每次买东西都回家取一次。

| 文件 | 作用 |
|------|------|
| user.js | 存储当前登录用户的信息（用户名、角色等） |
| theme.js | 存储当前的主题设置（浅色/深色） |

#### styles/ 文件夹 - "化妆间"

**作用：** 存放全局样式文件，定义页面的颜色、字体、间距等外观设置。

#### utils/ 文件夹 - "百宝箱"

**作用：** 存放通用的工具代码，供其他文件调用。

| 文件 | 作用 |
|------|------|
| request.js | 网络请求工具，封装了发送HTTP请求的通用逻辑，自动添加登录令牌等 |

#### views/ 文件夹 - "房间目录"

**作用：** 这里是系统的核心，包含了所有功能页面。每个子文件夹对应一个功能模块。

| 文件夹 | 对应功能 | 里面的文件 |

| club/ | 社团管理 | index.vue（社团列表）、member.vue（成员管理）、type.vue（类型管理） |

### 后端文件夹详解（backend/）

后端是系统的核心，负责处理所有业务逻辑和数据操作。用 Spring Boot 框架开发，采用经典的 MVC 分层架构。

```
backend/src/main/
├── java/com/club/
│   ├── ClubManagerApplication.java  ← 系统启动入口
│   ├── annotation/  ← 自定义注解（给代码加标签）
│   ├── aspect/      ← 切面（横切关注点）
│   ├── common/      ← 通用工具类
│   ├── config/      ← 配置类
│   ├── controller/  ← 控制器层（接收请求）
│   ├── dto/         ← 数据传输对象
│   ├── entity/      ← 实体类（对应数据库表）
│   ├── enums/       ← 枚举类（固定值定义）
│   ├── exception/   ← 异常处理
│   ├── mapper/      ← 数据访问层（操作数据库）
│   ├── security/    ← 安全认证
│   └── service/     ← 业务逻辑层
└── resources/
    └── application.yml  ← 项目配置文件
```

#### 分层架构说明

后端采用**分层架构**，像公司的组织结构一样，各层各司其职：

```
请求进来 → Controller（前台接待）→ Service（业务部门）→ Mapper（数据库部门）→ 数据库
                ↑                                              ↓
                ← ← ← ← ← ← ← ← 结果返回 ← ← ← ← ← ← ← ← ← ←
```

| 层级 | 类比 | 职责 | 文件示例 |
|------|------|------|----------|
| Controller | 前台接待 | 接收请求，转交给Service，返回结果 | AuthController, ClubController |
| Service | 业务部门 | 处理具体的业务逻辑 | ClubService, ActivityService |
| Mapper | 数据库部门 | 和数据库打交道，做增删改查 | ClubMapper, ActivityMapper |
| Entity | 数据模型 | 定义数据的结构（像表格的表头） | User, Club, Activity |

#### 各文件夹详解

**annotation/ - 自定义注解**

**aspect/ - 切面**

**common/ - 通用工具类**

| 文件 | 作用 |
|------|------|

**config/ - 配置类**

| 文件 | 作用 |
|------|------|

**controller/ - 控制器（接待请求的）**

每个Controller对应一个功能模块，负责接收前端发来的HTTP请求：

| 文件 | 管理什么 |

**dto/ - 数据传输对象**

DTO是在各层之间传递数据的"信封"，把前端提交的数据包装成规范格式：

| 文件 | 用途 |

## 四、技术选型说明

### 什么是技术选型？

技术选型就是"用什么工具来做这个系统"，就像装修房子要选材料一样。

### 前端技术

| 技术 | 是什么 | 为什么选它 |
|------|--------|------------|
| **Vue 3** | 前端框架，用来构建用户界面 | 国内最流行的前端框架之一，文档好、社区活跃、上手简单 |
| **Element Plus** | UI组件库，提供现成的按钮、表格、表单等 | 不用从零设计界面，有一套现成的漂亮组件可以直接用 |
| **Pinia** | 状态管理，存储全局数据 | 管理全局共享数据（如用户信息），比传统Vuex更简洁 |
| **Vite** | 构建工具，帮我们打包和开发 | 比Webpack快很多，开发体验好 |
| **ECharts** | 图表库，画统计图 | 数据可视化，让数据看板更好看 |
| **Axios** | HTTP请求库，发网络请求 | 和后端通信的工具 |

### 后端技术

| 技术 | 是什么 | 为什么选它 |
|------|--------|------------|
| **Spring Boot** | Java后端框架，快速搭建服务器 | 自动配置、生态完善、企业使用广泛 |
| **MyBatis-Plus** | 数据库操作工具，简化SQL编写 | 在MyBatis基础上增强，提供通用CRUD方法 |
| **Spring Security** | 安全框架，负责登录验证和权限控制 | 业界标准，功能强大，可定制性强 |
| **JWT** | 认证令牌，保持用户登录状态 | 无状态认证，适合前后端分离架构 |
| **MySQL** | 数据库，存储所有数据 | 开源免费、稳定可靠、社区活跃 |

### 数据库

| 技术 | 说明 |
|------|------|
| **MySQL** | 关系型数据库，存储结构化数据 |
| **13张表** | 覆盖用户、社团、活动、申请、通知、缴费等核心业务 |

## 五、系统功能详细说明

### 1. 用户管理模块

**功能列表：**
- 用户注册：填写学号、姓名、密码等信息注册账号

## 八、安全设计说明

### 密码安全

- 密码使用 **BCrypt算法** 加密存储，不是明文
- 即使数据库被泄露，攻击者也无法直接获取用户密码
- BCrypt自带随机盐值，同样的密码加密后结果不同

### 登录认证

- 采用 **JWT（JSON Web Token）** 机制
- 登录成功后服务器生成令牌，有效期24小时
- 后续请求携带令牌，服务器验证令牌后处理请求
- 无需在服务器端存储会话信息，支持水平扩展

### 权限控制

- 采用 **RBAC（基于角色的访问控制）** 模型
- 三种角色对应不同的操作权限
- 关键接口通过权限校验，无权限无法访问

### 数据安全

- 敏感数据传输使用HTTPS加密
- 用户输入经过校验，防止SQL注入和XSS攻击
- 系统日志记录所有操作，支持审计追溯
