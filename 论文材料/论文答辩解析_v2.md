## 一、选题背景与意义类问题

### 问题1：为什么选择这个课题？/ 研究背景是什么？

**回答要点：**

**（1）现实需求驱动**

高校学生社团是校园文化建设的重要载体。据统计，国内综合性大学普遍拥有数十个乃至上百个学生社团，涵盖学术研究、文艺创作、体育运动、志愿服务等多个领域。

然而，目前多数社团的日常管理仍存在以下问题：
- **信息管理碎片化**：社团信息分散在Excel表格、纸质文档和微信群中，缺乏统一管理平台，导致信息孤岛现象严重
- **审批流程低效化**：社团创建、活动审批等流程需要逐级上报，线下审批周期长、效率低下
- **数据统计滞后化**：社团运营数据（成员规模、活动参与度、经费使用等）缺乏实时统计与可视化分析手段
- **沟通协同困难化**：通知发布依赖社交平台，信息传达不及时、已读状态无法追踪

**（2）技术发展成熟**

Spring Boot和Vue.js等技术经过多年发展，生态完善、社区活跃，为快速开发Web应用提供了成熟的技术基础。前后端分离架构已成为主流开发模式，便于团队协作和后期维护。

**（3）信息化建设趋势**

高校信息化建设不断深入，教务管理系统、学生工作管理系统等已广泛部署。社团管理作为学生工作的重要组成部分，其信息化建设具有现实必要性。

- 增强用户体验，为学生提供便捷的一站式社团服务
- 为高校学生工作部门提供数据决策支持

### 问题3：国内外研究现状如何？

**回答要点：**

**国内现状：**
- **通用办公平台**：钉钉、企业微信等提供审批流程、通知推送等基础功能，但未针对高校社团场景进行深度适配。社团的分类分级管理、活动审批流程、社长角色权限配置等功能难以直接实现
- **校园信息化平台**：多数高校已部署教务管理系统、学生工作管理系统等，但社团管理模块功能简单，通常只是信息展示，缺乏全流程管理
- **开源项目**：部分高校学生以课程设计或毕业设计形式开发过社团管理系统，发布在GitHub等平台。但这些项目普遍存在功能模块不完善的问题，缺少活动全流程管理、缴费管理、通知推送等功能模块

**国外现状：**
- 国外高校多使用成熟的社团管理SaaS平台（如CampusLabs、Engage等），功能完善但不符合国内高校的管理习惯
- 国外高校社团管理更强调学生自治，审批流程相对简单

**结论：**
国内高校社团管理的信息化建设仍有较大的提升空间，开发一套功能完善、技术先进、操作便捷的社团管理平台具有研究价值和实践意义。

## 二、技术架构类问题

### 问题4：为什么选择 Spring Boot 框架？有什么优势？

**回答要点：**

**选择原因：**
1. **自动配置**：Spring Boot内置自动配置机制，能够根据项目引入的依赖自动完成大部分基础配置工作，大幅减少手动配置的工作量
2. **起步依赖**：通过starter依赖，将常用库的版本冲突问题交由Spring Boot管理，简化了依赖配置
3. **嵌入式服务器**：内置Tomcat服务器，可将应用打包为可执行的JAR文件，通过简单命令即可启动
4. **生态完善**：Spring Boot拥有庞大的生态系统，与MyBatis-Plus、Spring Security等框架无缝集成
5. **社区活跃**：国内开发者社区活跃，遇到问题容易找到解决方案
6. **企业应用广泛**：Spring Boot是国内企业级Java开发的事实标准，具有良好的就业前景

**与传统Spring的区别：**
- 传统Spring需要大量XML配置和注解配置
- Spring Boot通过自动配置和约定优于配置的理念，大幅简化了开发过程
- 传统Spring需要部署到外部应用服务器，Spring Boot可以直接打包运行

### 问题5：为什么选择 Vue.js 做前端？前后端分离有什么好处？

**回答要点：**

**选择Vue.js的原因：**
1. **轻量易上手**：Vue.js是一款轻量级框架，文档完善，学习曲线平缓
2. **声明式编程**：采用声明式的数据绑定机制，开发者只需关注数据和界面的映射关系，框架自动完成界面更新
3. **组件化开发**：支持将页面拆分为独立的组件，每个组件封装模板、逻辑和样式，便于复用和维护
4. **国内生态好**：Vue.js在国内开发者群体中使用广泛，Element Plus等优秀组件库基于Vue开发
5. **Vue 3 Composition API**：组合式API提供了更好的逻辑复用和代码组织方式

**前后端分离的好处：**
1. **职责分离**：前端专注于界面展示和用户交互，后端专注于业务逻辑和数据处理
2. **并行开发**：前后端可以同时开发，通过约定好的API接口进行联调
3. **技术独立**：前端和后端可以独立选择技术栈，后期也可以独立升级
4. **多端复用**：同一后端可以支持Web端、移动端、小程序等多个前端
5. **部署灵活**：前端可以部署在CDN上，后端可以部署在服务器上，各自独立扩展

### 问题7：什么是前后端分离架构？和传统架构有什么区别？

**回答要点：**

**传统架构（前后端耦合）：**
```
用户请求 → 服务器（Java渲染HTML页面）→ 返回完整HTML页面
```
- 服务器负责渲染页面，返回的是完整的HTML
- 前端代码和后端代码混合在一起
- 页面刷新需要重新加载整个页面

**前后端分离架构：**
```
用户请求 → 前端（Vue渲染页面）← API数据 → 后端（返回JSON数据）
```
- 前端负责页面渲染，后端只返回数据（JSON格式）
- 前端和后端是两个独立的应用，分别部署
- 页面局部更新，无需重新加载整个页面

**主要区别：**
| 对比项 | 传统架构 | 前后端分离 |
|--------|----------|------------|
| 数据格式 | HTML页面 | JSON数据 |
| 页面渲染 | 服务器端 | 浏览器端 |
| 通信方式 | 页面跳转 | API调用 |
| 部署方式 | 单体部署 | 独立部署 |

### 问题8：系统的整体架构是什么样的？

**回答要点：**

本系统采用**前后端分离的B/S架构**，整体分为三层：

```
┌─────────────────────────────────────────────────┐
│                  表现层（前端）                    │
│         Vue 3 + Element Plus + Pinia            │
│         负责页面展示、用户交互、数据渲染           │
└─────────────────────┬───────────────────────────┘
                      │ RESTful API（JSON）
┌─────────────────────▼───────────────────────────┐
│                  业务层（后端）                    │
│              Spring Boot 框架                     │
│  Controller → Service → Mapper                   │
│  负责请求处理、业务逻辑、权限控制                  │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│                  数据层（数据库）                  │
│                   MySQL                          │
│              负责数据的持久化存储                  │
└─────────────────────────────────────────────────┘
```

**后端分层架构：**
- **Controller层**：接收前端HTTP请求，解析请求参数，调用Service层处理业务，返回处理结果
- **Service层**：实现具体的业务逻辑，处理数据校验、业务规则等
- **Mapper层**：与数据库交互，执行增删改查操作
- **Entity层**：定义数据模型，与数据库表一一对应

### 问题11：什么是E-R图？能解释一下实体间的关系吗？

**回答要点：**

**E-R图定义：**
E-R图（Entity-Relationship Diagram，实体-关系图）是数据库概念设计的工具，用图形化方式展示实体、属性和实体之间的关系。

**E-R图符号说明：**
- **矩形**表示实体（如用户、社团、活动）
- **椭圆**表示属性（如用户名、社团名称）
- **菱形**表示关系（如"加入"、"发布"）
- **连线上的数字**表示基数（1表示一方，N表示多方）

**系统主要实体关系：**
- 用户（1）→ 加入 → 社团成员（N）
- 社团（1）→ 包含 → 社团成员（N）
- 社团（1）→ 发布 → 活动（N）
- 活动（1）→ 报名 → 报名记录（N）
- 用户（1）→ 参加 → 报名记录（N）
- 用户（1）→ 提交 → 入团申请（N）
- 社团（1）→ 接收 → 入团申请（N）
- 用户（1）→ 发送 → 通知（N）
- 用户（1）→ 阅读 → 通知阅读记录（N）
- 社团（1）→ 发布 → 缴费通知（N）

### 问题12：为什么有些表用了冗余字段（如president_name）？这不是违反范式吗？

**回答要点：**

**这是有意为之的反范式设计，属于以空间换时间的优化策略。**

**原因说明：**
1. 社团表中的`president_name`、活动表中的`club_name`等字段确实与主表存在数据冗余
2. 这些字段是高频查询字段，如果不冗余存储，每次查询都需要多表关联
3. 多表关联操作在数据量较大时会显著影响查询性能

**权衡取舍：**
- **优点**：减少JOIN操作，提升查询性能
- **缺点**：增加存储空间，需要在数据变更时同步更新冗余字段
- **保障机制**：应用层负责在数据变更时同步更新冗余字段，确保数据一致性

**答辩时的表述：**
"这是反范式设计，属于以空间换时间的优化策略。冗余字段用于减少高频查询时的多表关联操作，应用层负责在数据变更时同步更新冗余字段。"

### 问题13：有些外键没有声明FOREIGN KEY约束，为什么？

**回答要点：**

**这是有意为之的设计选择。**

1. 系统在**应用层**保证数据一致性，通过业务逻辑校验数据的有效性
2. 数据库层未设置外键约束，以避免写入时的**级联检查开销**
3. 在高并发场景下，外键约束可能导致死锁和性能下降
4. 系统通过**统一的Service层**操作数据，所有写入操作都经过业务校验

**答辩时的表述：**
"系统在应用层保证数据一致性，数据库层未设置外键约束以避免写入时的级联检查开销，提升写入性能。"

### 问题14：什么是RBAC权限模型？系统是怎么实现的？

**回答要点：**

**RBAC定义：**
RBAC（Role-Based Access Control，基于角色的访问控制）是一种权限管理模型，通过角色来关联用户和权限。用户不直接拥有权限，而是通过角色间接获得权限。

**RBAC三要素：**
- **用户（User）**：系统的使用者
- **角色（Role）**：权限的集合，如管理员、社长、普通用户
- **权限（Permission）**：对资源的操作许可

**本系统的RBAC实现：**
1. 用户表中的`role`字段定义角色（1=管理员、2=社长、3=普通用户）
2. Spring Security根据用户角色加载对应的权限集合
3. 接口通过`@PreAuthorize`注解配置所需的权限
4. 请求到达时，Spring Security校验用户是否具备相应权限

## 四、安全相关问题

### 问题16：系统是怎么保证安全性的？

**回答要点：**

系统从以下四个方面保障安全性：

**（1）身份认证**
- 采用JWT令牌认证机制
- 用户登录后获取令牌，后续请求携带令牌证明身份
- 令牌设置有效期（24小时），过期需重新登录

**（2）访问控制**
- 基于RBAC模型实施差异化权限管理
- 关键接口配置权限要求，未授权无法访问
- 不同角色看到不同的菜单和功能

**（3）数据加密**
- 密码采用BCrypt算法不可逆加密存储
- BCrypt自带随机盐值，防止彩虹表攻击
- 敏感数据传输使用HTTPS加密

**（4）操作审计**
- 通过AOP切面自动记录用户操作日志
- 日志包含操作人、操作时间、操作内容、IP地址等
- 支持异常行为追踪和审计

### 问题17：JWT是什么？工作原理是什么？

**回答要点：**

**JWT定义：**
JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在各方之间安全地传递信息。JWT由三部分组成：

**JWT结构：**
```
Header.Payload.Signature
```
- **Header（头部）**：包含令牌类型和加密算法信息
- **Payload（载荷）**：包含用户标识、过期时间等声明信息
- **Signature（签名）**：用于验证令牌的完整性

**工作流程：**
1. 用户登录，服务器验证用户名密码
2. 验证通过后，服务器根据用户信息生成JWT令牌
3. 服务器将令牌返回给前端
4. 前端将令牌存储在本地（localStorage）
5. 后续请求在请求头中携带该令牌
6. 服务器验证令牌签名和有效期后处理请求

**JWT的优点：**
- **无状态**：服务器无需存储会话信息
- **可扩展**：适合分布式系统和微服务架构
- **跨域友好**：天然支持跨域请求

### 问题18：密码是怎么存储的？为什么不用明文？

**回答要点：**

**密码存储方式：**
- 密码使用**BCrypt算法**进行不可逆加密后存储
- BCrypt是一种专为密码设计的哈希算法，具有以下特点：

**BCrypt的特点：**
1. **不可逆**：加密后无法还原为原始密码
2. **自带盐值**：每次加密会自动生成随机盐值，即使两个用户密码相同，加密后的结果也不同
3. **计算成本可调**：可以通过work factor调整计算成本，抵御暴力破解

**为什么不用明文存储：**
1. 明文存储一旦数据库泄露，所有用户密码将暴露
2. 用户往往在多个平台使用相同密码，泄露风险波及范围广
3. 法律要求对用户密码进行加密存储

### 问题19：Spring Security是怎么工作的？

**回答要点：**

**Spring Security工作原理：**

Spring Security通过**过滤器链（Filter Chain）**机制工作。当请求到达时，会依次经过一系列安全过滤器：

```
HTTP请求 → CorsFilter → JwtAuthenticationFilter → AuthorizationFilter → Controller
```

**核心组件：**

1. **JwtAuthenticationFilter（认证过滤器）**
   - 从请求头中提取JWT令牌
   - 验证令牌的有效性
   - 解析用户信息并设置到SecurityContext中

2. **JwtAuthenticationEntryPoint（认证入口）**
   - 当用户未认证或令牌失效时触发
   - 返回401未认证的错误信息

3. **UserDetailsServiceImpl（用户详情服务）**
   - 根据用户名从数据库加载用户信息
   - 将用户信息封装为UserDetails对象供框架使用

4. **SecurityConfig（安全配置）**
   - 配置哪些接口需要认证
   - 配置权限要求
   - 注册自定义过滤器

### 问题20：MyBatis-Plus和MyBatis有什么区别？

**回答要点：**

**MyBatis：**
- 一款优秀的持久层框架，通过XML或注解编写SQL
- 灵活性高，适合复杂的SQL查询
- 单表操作需要手动编写SQL

**MyBatis-Plus：**
- MyBatis的增强工具，在MyBatis基础上只做增强不做改变
- 内置通用Mapper，提供CRUD方法，无需编写单表SQL
- 提供条件构造器Wrapper，支持链式调用构建查询条件
- 支持分页插件、代码生成器、自动填充、逻辑删除等实用功能

**对比：**
| 对比项 | MyBatis | MyBatis-Plus |
|--------|---------|--------------|
| 单表操作 | 需手写SQL | 自动生成 |
| 复杂查询 | XML配置 | XML配置 + Wrapper |
| 分页 | 手动实现 | 内置插件 |
| 代码量 | 较多 | 大幅减少 |

### 问题21：什么是RESTful API？系统是怎么设计接口的？

**回答要点：**

**RESTful定义：**
REST（Representational State Transfer）是一种API设计风格，RESTful API是遵循REST原则的接口设计。

**RESTful设计原则：**
1. **资源定位**：URL表示资源，使用名词而非动词
   - ✅ `GET /api/club/1` （获取ID为1的社团）
   - ❌ `GET /api/getClubById?id=1`
2. **HTTP方法表示操作**
   - GET：查询
   - POST：新增
   - PUT：更新
   - DELETE：删除
3. **返回JSON格式数据**

**系统接口设计示例：**
| 操作 | 方法 | URL | 说明 |
|------|------|-----|------|
| 查询社团列表 | GET | /api/club | 获取社团列表 |
| 获取社团详情 | GET | /api/club/{id} | 获取单个社团 |
| 创建社团 | POST | /api/club | 提交创建申请 |
| 修改社团 | PUT | /api/club/{id} | 更新社团信息 |
| 删除社团 | DELETE | /api/club/{id} | 注销社团 |

### 问题22：Vue 3的组合式API和选项式API有什么区别？

**回答要点：**

**选项式API（Options API）- Vue 2风格：**
```javascript
export default {
  data() { return { count: 0 } },
  methods: { increment() { this.count++ } },
  computed: { doubleCount() { return this.count * 2 } }
}
```
- 按选项（data、methods、computed）组织代码
- 逻辑分散在不同选项中，相关代码可能相距较远
- 适合小型项目

**组合式API（Composition API）- Vue 3特色：**
```javascript
import { ref, computed } from 'vue'
export default {
  setup() {
    const count = ref(0)
    const increment = () => count.value++
    const doubleCount = computed(() => count.value * 2)
    return { count, increment, doubleCount }
  }
}
```
- 按逻辑功能组织代码
- 相关逻辑放在一起，便于维护和复用
- 适合大型项目和逻辑复用场景

**本系统使用组合式API的原因：**
- 项目功能模块较多，需要良好的代码组织
- 抽取了useDialog、usePagination等可复用的组合式函数

## 六、系统实现类问题

### 问题24：用户登录流程是怎样的？

**回答要点：**

**登录流程：**
1. 用户在登录页面输入用户名和密码
2. 前端通过auth.js发送POST请求到 `/api/auth/login`
3. 后端AuthController接收请求，调用Spring Security进行认证
4. Security验证用户名和密码（BCrypt比对）
5. 验证通过后，JwtUtils生成JWT令牌
6. 后端将令牌返回给前端
7. 前端将令牌存储到localStorage
8. 后续请求在请求头中携带 `Authorization: Bearer {token}`

---

### 问题25：社团创建的流程是怎样的？

**回答要点：**

**创建流程：**
1. 用户登录后，点击"创建社团"
2. 填写社团名称、类型、描述等信息，提交申请
3. 系统将申请记录存入sys_club表，状态设为"待审核（0）"
4. 管理员在审核页面看到待审核的社团列表
5. 管理员点击"通过"或"拒绝"
   - 通过：状态更新为"正常运营（1）"，社团正式成立
   - 拒绝：状态更新为"已拒绝"，记录拒绝原因
6. 系统通知申请者审核结果

### 问题26：活动发布需要审核吗？流程是什么？

**回答要点：**

**需要审核。**

**发布流程：**
1. 社长创建活动，填写活动标题、描述、时间、地点、人数限制等信息
2. 提交后活动状态设为"待审核（1）"
3. 管理员对活动信息进行审核
4. 审核通过后，活动状态更新为"已发布（2）"，学生可以报名
5. 审核不通过则返回修改意见，社长修改后重新提交

**报名流程：**
1. 普通用户浏览已发布的活动
2. 选择感兴趣的活动，点击"报名"
3. 报名记录存入活动报名表，状态为"待审核"
4. 社长对报名信息进行审核（可以批量操作）
5. 审核通过后，用户可以参加活动
6. 活动开始后，参与者可以进行签到

### 问题27：通知是怎么推送给用户的？

**回答要点：**

**通知类型：**
- **系统通知**：管理员发布，面向全体用户
- **社团通知**：社长发布，面向本社团成员

**推送流程：**
1. 发布者填写通知标题、内容、类型、优先级
2. 系统将通知存入sys_notification表
3. 如果是社团通知，系统自动向该社团所有成员的sys_notification_read表插入记录
4. 如果是系统通知，系统向所有用户的sys_notification_read表插入记录
5. 用户登录后，在通知页面查看未读通知
6. 用户点击通知后，系统更新对应记录的已读状态和阅读时间

---

### 问题28：社长转移是怎么实现的？

**回答要点：**

**转移流程：**
1. 现任社长发起转移请求，选择接收方成员
2. 系统校验接收方是否为本社团成员
3. 转移记录存入sys_president_transfer表，状态为"待确认"
4. 接收方确认接收社长职位
5. 系统更新社团表的president_id字段
6. 转移记录状态更新为"已完成"
7. 原社长降为普通成员，新社长获得管理权限
