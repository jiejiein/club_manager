## 一、选题背景与研究意义

### 1.1 研究背景

随着我国高等教育事业的蓬勃发展，高校学生社团作为校园文化建设的重要载体，其规模与数量呈逐年递增趋势。据相关统计，国内综合性大学普遍拥有百余个学生社团，涵盖学术科技、文化艺术、体育竞技、社会实践及兴趣爱好五大类别。然而，当前多数高校社团管理仍依赖传统的人工方式，存在以下突出问题：

1. **信息管理碎片化**：社团信息分散于Excel表格、纸质文档及即时通讯群组中，缺乏统一管理平台，导致信息孤岛现象严重。
2. **审批流程低效化**：社团创建、活动申报、入团申请等流程需逐级呈报，线下审批周期长、效率低下。
3. **数据统计滞后化**：社团运营数据（成员规模、活动参与度、经费使用等）缺乏实时统计与可视化分析手段。
4. **沟通协同困难化**：通知发布依赖社交平台，信息传达不及时、已读状态无法追踪。

### 1.2 研究意义

本课题针对上述痛点，设计并实现一套基于B/S架构的高校社团管理与活动预约平台，具有以下理论与实践意义：

**理论意义：**
- 探索RBAC（基于角色的访问控制）模型在多层级组织管理系统中的应用方法
- 研究前后端分离架构在教育信息化系统中的设计模式与实现方案
- 为高校信息化管理提供可复用的技术参考架构

**实践意义：**
- 提升社团管理效率，实现业务流程的规范化与自动化
- 降低管理成本，减少人工操作导致的信息误差
- 增强用户体验，为学生提供便捷的一站式社团服务
- 为高校学生工作部门提供数据决策支持

### 1.3 国内外研究现状

**国外现状：** 欧美高校普遍采用成熟的校园管理系统（如CampusLabs、OrgSync等），功能完善但多为商业化付费产品，且存在本土化适配问题。

**国内现状：** 国内高校信息化建设起步较晚，现有社团管理系统多为定制开发，存在技术栈陈旧、功能单一、扩展性差等问题。基于SpringBoot + Vue3前后端分离架构的社团管理系统研究尚处于发展阶段，具有一定的研究空间。

## 二、系统需求分析

### 2.1 功能需求分析

通过对目标用户（高校社团管理者、社团负责人、普通学生）的调研分析，系统功能需求归纳如下：

#### 2.1.1 用户管理需求
- 支持用户注册、登录、信息维护等基础功能
- 支持基于角色的权限分级控制（管理员、社长、普通用户）
- 支持用户状态管理（启用/禁用）与账号安全管理

#### 2.1.2 社团管理需求
- 支持社团创建申请与管理员审核机制
- 支持社团信息的增删改查及状态管理
- 支持社团类型分类管理（学术科技类、文化艺术类、体育竞技类、社会实践类、兴趣爱好类）

#### 2.1.3 活动管理需求
- 支持活动的创建、编辑、发布、取消等全生命周期管理
- 支持活动报名审核与参与人数控制
- 支持活动签到与评价反馈机制

#### 2.1.4 申请审批需求
- 支持入团申请的提交、审批、驳回流程
- 支持批量审批操作
- 支持申请状态的实时跟踪

#### 2.1.5 通知管理需求
- 支持系统级通知与社团级通知的分级发布
- 支持通知优先级设置（普通、重要、紧急）
- 支持通知已读/未读状态追踪

#### 2.1.6 缴费管理需求
- 支持缴费通知的发布与管理
- 支持缴费状态跟踪与逾期提醒
- 支持缴费记录的统计查询

#### 2.1.7 数据看板需求
- 支持社团数量、成员规模、活动数据的统计分析
- 支持数据可视化图表展示（饼图、柱状图、折线图）

### 2.2 非功能需求分析

| 需求类型 | 具体要求 |
|---------|---------|
| **性能需求** | 系统响应时间 ≤ 2秒，支持100+并发用户 |
| **安全需求** | 采用JWT令牌认证，密码加密存储，防止SQL注入与XSS攻击 |
| **可用性需求** | 界面简洁友好，支持响应式布局，兼容主流浏览器 |
| **可扩展性** | 采用模块化设计，便于功能扩展与维护 |
| **可靠性需求** | 系统可用性 ≥ 99%，数据定期备份 |

### 2.3 用例分析

系统共涉及三个角色，核心用例如下：

| 角色 | 用例描述 |
|------|---------|
| **管理员** | 用户管理、社团审核、类型管理、活动审核、通知发布、数据统计 |
| **社长** | 成员管理、活动发布与管理、通知发布、缴费管理、社团数据查看 |
| **普通用户** | 浏览社团、申请入团、活动报名、通知查看、费用缴纳、个人信息管理 |

## 三、系统总体设计

### 3.1 系统架构设计

本系统采用**前后端分离**的B/S架构，整体架构分为三层：

```
┌─────────────────────────────────────────────────────┐
│                    表现层 (Presentation Layer)        │
│            Vue 3 + Element Plus + ECharts           │
│         用户界面、交互逻辑、数据可视化展示              │
├─────────────────────────────────────────────────────┤
│                    业务逻辑层 (Business Layer)        │
│           SpringBoot + Spring Security + JWT        │
│         业务处理、权限控制、接口服务、数据校验           │
├─────────────────────────────────────────────────────┤
│                    数据访问层 (Data Access Layer)     │
│               MyBatis-Plus + MySQL 8.0              │
│         数据持久化、SQL映射、数据库连接管理             │
└─────────────────────────────────────────────────────┘
```

**架构优势：**
1. **前后端解耦**：前后端独立开发、独立部署，通过RESTful API通信
2. **职责清晰**：各层分工明确，降低系统耦合度，便于维护与扩展
3. **技术灵活**：前端可独立替换技术栈，后端可独立进行性能优化

## 四、系统详细设计

### 4.1 权限控制设计

本系统采用**RBAC（Role-Based Access Control）模型**实现权限管理，权限分配逻辑如下：

| 角色 | 权限范围 | 访问控制 |
|------|---------|---------|
| 管理员（role=1） | 全平台数据读写权限 | 可访问所有API接口 |
| 社长（role=2） | 仅限所管辖社团的数据操作 | 仅可操作本社团相关资源 |
| 普通用户（role=3） | 个人数据读写 + 公共信息查看 | 仅可操作个人相关资源 |

**技术实现：**
- 采用Spring Security + JWT实现认证与授权
- 自定义注解实现接口级别的权限校验
- 密码采用BCrypt算法加密存储

### 4.2 核心业务流程设计

#### 4.2.1 社团创建流程

```
用户提交创建申请 → 系统校验信息完整性 → 写入数据库(状态:待审核)
                                              ↓
管理员审核社团 ← 通知管理员审核 ← 数据更新
      ↓                              ↓
  通过/拒绝 → 更新状态 → 通知申请人 → 流程结束
```

#### 4.2.2 活动管理流程

```
社长创建活动(草稿) → 提交审核 → 管理员审核
                              ↓
                    通过 → 状态更新为"已发布" → 用户可报名
                    拒绝 → 填写拒绝原因 → 社长修改后重新提交
                                              ↓
                              活动当天 → 用户签到 → 活动结束 → 评价反馈
```

#### 4.2.3 入团申请流程

```
用户浏览社团 → 提交入团申请(填写理由) → 数据写入申请表
                                              ↓
                              社长收到审核通知 → 审核处理
                                    ↓
                        通过 → 更新成员表 → 通知用户
                        拒绝 → 填写拒绝原因 → 通知用户
```

### 4.3 状态机设计

**活动状态流转：**

```
[草稿] --提交--> [待审核] --通过--> [已发布] --开始--> [进行中] --结束--> [已结束]
                    |                  |
                    +----拒绝----+     +---取消---+
                                   ↓         ↓
                              [已取消]    [已取消]
```

**社团状态流转：**

```
[待审核] --通过--> [正常运营] --注销--> [已注销]
    |
    +--拒绝--> [已拒绝]
```

### 4.4 接口设计

系统采用**RESTful API**设计规范，统一返回格式如下：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": { }
}
```

**主要API接口：**

| 模块 | 接口前缀 | 主要接口 |
|------|---------|---------|
| 认证 | /api/auth | 登录、注册、Token刷新 |
| 用户 | /api/user | 信息查询、信息修改、状态管理 |
| 社团 | /api/club | CRUD操作、审核、成员管理 |
| 活动 | /api/activity | CRUD操作、报名、签到 |
| 申请 | /api/apply | 提交、审批、查询 |
| 通知 | /api/notification | 发布、列表、已读标记 |
| 缴费 | /api/payment | 发布、查询、缴费操作 |
| 看板 | /api/dashboard | 统计数据查询 |

## 五、系统实现

### 5.1 后端实现

后端采用**分层架构**设计，各层职责如下：

| 层级 | 包名 | 职责 |
|------|------|------|
| 控制层 | controller | 接收HTTP请求，参数校验，调用服务层 |
| 服务层 | service | 业务逻辑处理，事务管理 |
| 数据访问层 | mapper | 数据库操作，SQL映射 |
| 实体层 | entity | 数据库实体类，与表结构一一对应 |
| 数据传输层 | dto | 前端请求参数封装 |
| 视图层 | vo | 前端响应数据封装 |
| 安全层 | security | 认证授权，JWT生成与验证 |
| 配置层 | config | 系统配置类 |
| 异常处理层 | exception | 全局异常捕获与处理 |

### 5.2 前端实现

前端采用**Vue 3 + Composition API**开发，核心模块设计：

| 模块 | 技术方案 |
|------|---------|
| 路由管理 | Vue Router 4，基于角色的路由守卫 |
| 状态管理 | Pinia，模块化Store设计 |
| UI组件 | Element Plus，按需引入 |
| HTTP请求 | Axios，请求/响应拦截器封装 |
| 数据可视化 | ECharts 5，图表组件封装 |
| 权限控制 | 路由守卫 + 按钮级权限指令 |

### 5.3 关键技术实现

#### 5.3.1 JWT认证机制

```
用户登录 → 验证凭证 → 生成JWT令牌 → 前端存储( localStorage )
                                          ↓
                                   每次请求携带Token
                                          ↓
                                   后端验证Token → 有效则放行
                                          ↓
                                   过期则返回401 → 刷新或重新登录
```

#### 5.3.2 全局异常处理

采用@ControllerAdvice + @ExceptionHandler实现统一异常响应，保证接口返回格式一致性。

#### 5.3.3 分页查询

采用MyBatis-Plus分页插件实现高效分页，避免全表扫描导致的性能问题。

## 七、系统特色与创新

### 7.1 多层级权限控制体系

基于RBAC模型实现细粒度的权限管理，支持**接口级**与**按钮级**双重权限控制，确保不同角色用户的操作安全性。

### 7.2 全生命周期审批流

社团创建、活动发布、入团申请等核心业务均采用**状态机驱动**的审批流程，实现业务流程的规范化与可追溯性。

### 7.3 数据可视化决策支持

集成ECharts实现多维度数据统计与可视化展示，为管理决策提供直观的数据支撑。

### 7.4 前后端分离架构

采用前后端完全分离的架构设计，实现**并行开发**、**独立部署**，显著提升开发效率与系统可维护性。
