## 一、这个系统是干什么的？

### 一句话概括

**这是一个"网上社团管理平台"，类似一个小型的"社团版微信小程序"。**

想象一下大学里的场景：

- 你是个学生，想加入吉他社，但不知道怎么报名
- 社长要统计谁交了社费，谁没交，谁参加了活动，一堆表格很头疼
- 学校管社团的老师要审核每个社团是否合规，看看整体数据

**这个系统就是为了解决这些问题而开发的。**

### 生活化比喻

| 系统功能 | 生活中的类比 |
|---------|------------|
| 用户注册/登录 | 就像注册微信账号 |
| 申请加入社团 | 就像在微信群里申请入群 |
| 社长审批 | 就像群主点"同意"让某人进群 |
| 发布活动 | 就像在朋友圈发活动通知 |
| 活动报名 | 就像点赞/评论表示"我要参加" |
| 社费缴纳 | 就像微信群里的"收款"功能 |
| 数据看板 | 就像微信后台的数据统计报表 |

## 二、系统里有哪几种人？（角色）

系统设计了 **三种角色**，每种角色能做的事情不一样：

### 1. 👑 管理员（超级大管家）

> 比如：学校社团联合会的老师

**能做的事：**
- 看到**所有**社团和**所有**用户
- 可以禁用某个捣乱的用户账号
- 审核社团是否允许成立
- 发布全校都能看到的通知
- 看到整体的数据统计（多少社团、多少人、多少活动）

### 2. 👤 社长（社团的小管家）

> 比如：吉他社社长、篮球社社长

**能做的事：**
- 管理自己社团的成员（同意谁加入、踢出谁）
- 发布自己社团的活动
- 审核报名参加活动的人
- 给社团成员发通知
- 发布社费缴费通知
- 看到自己社团的数据统计

### 3. 👨‍🎓 普通用户（学生）

> 比如：想加入社团的普通同学

**能做的事：**
- 浏览有哪些社团
- 申请加入感兴趣的社团
- 报名参加社团活动
- 签到参加活动
- 查看通知
- 在线缴纳社费
- 给活动打分评价

## 三、系统有哪些核心功能？

### 📁 功能模块一览

```
┌─────────────────────────────────────────────────┐
│              高校社团管理平台                      │
├─────────────┬─────────────┬─────────────────────┤
│  用户管理    │  社团管理    │  活动管理            │
├─────────────┼─────────────┼─────────────────────┤
│  申请管理    │  通知管理    │  缴费管理            │
├─────────────┴─────────────┴─────────────────────┤
│              数据看板（统计报表）                   │
└─────────────────────────────────────────────────┘
```

### 各模块详细说明

#### 1. 用户管理 👥

**做什么用的？** —— 管理"谁可以用这个系统"

| 功能 | 说明 |
|------|------|
| 注册 | 学生用自己的学号、姓名注册账号 |
| 登录 | 输入用户名密码进入系统 |
| 修改个人信息 | 修改昵称、手机号、头像等 |
| 用户状态管理 | 管理员可以把违规用户"关掉"（禁用） |
| 角色分配 | 管理员可以把普通用户升级为社长 |

#### 2. 社团管理 🏛️

**做什么用的？** —— 管理"有哪些社团"

| 功能 | 说明 |
|------|------|
| 创建社团 | 学生提交社团创建申请 |
| 社团审核 | 管理员审核是否允许成立 |
| 社团列表 | 查看所有社团（可按类型筛选） |
| 社团详情 | 查看社团介绍、成员列表、活动列表 |
| 社团注销 | 不再运营的社团可以被注销 |
| 社团类型 | 管理员维护社团分类（学术类、文艺类、体育类等） |

#### 3. 活动管理 📋

**做什么用的？** —— 管理"社团举办的各种活动"

| 功能 | 说明 |
|------|------|
| 发布活动 | 社长填写活动标题、时间、地点、人数限制等 |
| 活动审核 | 需要管理员审核后才能对外发布 |
| 活动报名 | 学生浏览活动并报名参加 |
| 报名审核 | 社长可以审核报名的人 |
| 活动签到 | 到了活动现场扫码签到 |
| 活动评价 | 活动结束后可以打分和写评价 |
| 活动状态 | 草稿 → 待审核 → 已发布 → 进行中 → 已结束 |

#### 4. 入团申请管理 ✋

**做什么用的？** —— 管理"谁想加入社团"

| 功能 | 说明 |
|------|------|
| 提交申请 | 学生选择社团，写上"我想加入的原因" |
| 审核处理 | 社长看到申请，点"同意"或"拒绝" |
| 批量操作 | 一次可以同意/拒绝多个人 |
| 申请记录 | 可以看到自己的申请处理到哪一步了 |

#### 5. 通知管理 📢

**做什么用的？** —— "消息推送"功能

| 功能 | 说明 |
|------|------|
| 系统通知 | 管理员发的，所有人都能看到 |
| 社团通知 | 社长发的，只有本社团成员能看到 |
| 优先级 | 普通 / 重要 / 紧急（紧急的会特别标注） |
| 已读状态 | 可以看到谁看了通知、谁没看 |

#### 6. 缴费管理 💰

**做什么用的？** —— 管理"社费缴纳"

| 功能 | 说明 |
|------|------|
| 发布缴费通知 | 社长设定金额、截止日期 |
| 在线缴费 | 学生点击"缴费"完成支付 |
| 缴费记录 | 查看谁交了、谁没交、谁逾期了 |
| 逾期提醒 | 过了截止日期的自动标记为"已逾期" |

#### 7. 数据看板 📊

**做什么用的？** —— "数据报表"，用图表展示信息

| 数据 | 说明 |
|------|------|
| 社团总数 | 全校共有多少个社团 |
| 成员统计 | 各社团分别有多少人 |
| 活动统计 | 本月/本学期共举办了多少活动 |
| 报名趋势 | 哪个活动最受欢迎 |
| 缴费统计 | 社费收缴情况 |

## 四、系统背后的"数据"是怎么存储的？

> 这部分用最简单的方式解释数据库里的每张表

### 数据库表总览（11张表）

想象数据库就像一个巨大的 **Excel 文件**，每张表就是一个 **Sheet 页**：

```
📊 数据库：club_manager（社团管理）

├── 👤 sys_user               = "用户表"（所有人的账号信息）
├── 📁 sys_club_type          = "社团类型表"（分类标签）
├── 🏛️ sys_club               = "社团表"（每个社团的基本信息）
├── 👥 sys_club_member        = "社团成员表"（谁在哪个社团）
├── 📋 sys_activity           = "活动表"（每个活动的详细信息）
├── ✅ sys_activity_sign_up   = "活动报名表"（谁报名了哪个活动）
├── 📝 sys_apply              = "入团申请表"（谁申请加入哪个社团）
├── 📢 sys_notification       = "通知表"（所有通知内容）
├── 👁️ sys_notification_read  = "通知阅读记录"（谁看了通知）
├── 💵 sys_payment            = "缴费通知表"（社长发布的缴费要求）
└── 💳 sys_payment_record     = "缴费记录表"（每个人的缴费情况）
```

### 每张表详细说明

#### 1. 用户表 `sys_user`

就像 **通讯录**，记录每个人的基本信息：

| 字段 | 含义 | 举例 |
|------|------|------|
| id | 每个人的唯一编号 | 1、2、3... |
| username | 登录用户名 | jiejie |
| password | 登录密码 | （加密存储） |
| nickname | 昵称 | 小杰 |
| role | 角色 | 1=管理员, 2=社长, 3=普通用户 |
| status | 账号状态 | 0=禁用, 1=正常 |
| student_no | 学号 | 2024001 |
| college | 所属学院 | 计算机学院 |
| major | 专业 | 软件工程 |

#### 2. 社团类型表 `sys_club_type`

就像 **文件夹的分类标签**：

| 类型名称 | 说明 |
|---------|------|
| 学术科技类 | 编程社、机器人社等 |
| 文化艺术类 | 书法社、话剧社等 |
| 体育竞技类 | 篮球社、足球社等 |
| 社会实践类 | 志愿者协会等 |
| 兴趣爱好类 | 摄影社、美食社等 |

#### 3. 社团表 `sys_club`

就像每个社团的 **"身份证"**：

| 字段 | 含义 |
|------|------|
| name | 社团名称 |
| type_name | 属于哪个类型 |
| president_name | 社长是谁 |
| description | 社团介绍 |
| status | 0=待审核, 1=正常运营, 2=已注销 |
| member_count | 当前有多少成员 |

#### 4. 社团成员表 `sys_club_member`

记录 **"谁在哪个社团"**，就像微信群的成员列表：

```
一个人可以加入多个社团，一个社团可以有多个成员
     （多对多关系）
```

#### 5. 活动表 `sys_activity`

记录每个活动的 **详细信息**：

| 字段 | 含义 |
|------|------|
| title | 活动标题 |
| club_name | 哪个社团办的 |
| location | 在哪举办 |
| start_time / end_time | 活动开始和结束时间 |
| max_participants | 最多能来多少人 |
| current_participants | 已经报了多少人 |
| status | 活动当前处于哪个阶段 |

#### 6. 活动报名表 `sys_activity_sign_up`

记录 **"谁报了哪个活动"**：

```
一个人可以报名多个活动，一个活动可以被多个人报名
     （多对多关系）
```

还包含：签到时间、评分、评价内容

#### 7. 入团申请表 `sys_apply`

记录 **"谁申请加入哪个社团"**：

| 字段 | 含义 |
|------|------|
| club_name | 想加入的社团 |
| user_name | 申请人的名字 |
| reason | 为什么想加入 |
| status | 0=待审核, 1=已通过, 2=已拒绝 |

#### 8. 通知表 `sys_notification`

记录 **所有通知内容**：

| 字段 | 含义 |
|------|------|
| title | 通知标题 |
| content | 通知内容 |
| type | 1=系统通知(所有人可见), 2=社团通知(仅成员可见) |
| priority | 0=普通, 1=重要, 2=紧急 |
| sender_name | 谁发的 |

#### 9. 通知阅读记录表 `sys_notification_read`

记录 **"谁看了哪个通知"**：

```
一条通知可以被很多人阅读，一个人可以阅读很多通知
     （多对多关系）
```

#### 10. 缴费通知表 `sys_payment`

社长发布的 **"收费要求"**：

| 字段 | 含义 |
|------|------|
| title | 缴费标题（如"2024年春季社费"） |
| amount | 金额（如 50.00 元） |
| deadline | 截止日期 |

#### 11. 缴费记录表 `sys_payment_record`

每个人的 **"缴费详情"**：

| 字段 | 含义 |
|------|------|
| user_name | 谁交的 |
| amount | 交了多少 |
| status | 0=未缴费, 1=已缴费, 2=已逾期 |
| pay_time | 什么时候交的 |
| transaction_no | 交易流水号 |

### 关系说明

| 关系 | 通俗解释 |
|------|---------|
| 用户 ──n:n── 社团 | 一个人可以加入多个社团，一个社团可以有多个成员 |
| 社团 ──1:n── 活动 | 一个社团可以举办很多活动，但一个活动只属于一个社团 |
| 用户 ──n:n── 活动 | 一个人可以报名多个活动，一个活动可以有多个人参加 |
| 社团 ──1:n── 缴费通知 | 一个社团可以发布多个缴费要求 |
| 通知 ──1:n── 阅读记录 | 一条通知可以被很多人阅读 |

## 六、技术架构（简单版）

> "系统是怎么跑起来的"

### 整体架构

```
    📱 用户的浏览器（你看到的页面）
           │
           ▼
   ┌─────────────────┐
   │   前端 (Vue3)    │  ← 你看到的界面、按钮、表格
   │   运行在 3000端口  │
   └────────┬────────┘
            │ 通过网络请求 (HTTP)
            ▼
   ┌─────────────────┐
   │   后端 (Java)     │  ← 处理业务逻辑、判断权限
   │   运行在 8080端口  │
   └────────┬────────┘
            │ 通过SQL查询
            ▼
   ┌─────────────────┐
   │   数据库 (MySQL)  │  ← 存储所有数据
   │   运行在 3306端口  │
   └─────────────────┘
```

### 三层架构的比喻

| 层级 | 比喻 | 作用 |
|------|------|------|
| 前端 | 餐厅的 **前台和菜单** | 顾客点菜、看菜单 |
| 后端 | 餐厅的 **后厨** | 做菜、处理订单 |
| 数据库 | 餐厅的 **仓库** | 存放食材、调料 |

顾客（用户）通过菜单（前端）点菜 → 服务员把单子给后厨（后端）→ 后厨从仓库（数据库）取食材做菜 → 菜做好了送到前台 → 顾客拿到菜

### 技术名词解释

| 技术 | 一句话解释 | 不懂也没关系 |
|------|-----------|-------------|
| **SpringBoot** | Java的开发框架，让后端开发更快 | 就像"厨房的标准化流程" |
| **Vue3** | 前端框架，让页面更流畅 | 就像"装修厨房的工具" |
| **MyBatis-Plus** | 帮后端跟数据库对话的工具 | 就像"采购员"，帮厨房从仓库取东西 |
| **MySQL** | 存数据的数据库系统 | 就像"仓库管理员" |
| **JWT** | 用户登录后给你一个"门票" | 就像"入场券"，证明你已登录 |
| **Element Plus** | 好看的界面组件库 | 就像"精美的餐具" |
| **ECharts** | 画统计图表的工具 | 就像"数据可视化画板" |
| **Pinia** | 管理前端临时数据的工具 | 就像"服务员的记事本" |
| **Vite** | 前端的构建工具 | 就像"厨房的电器" |

```
        └── user/              ←   普通用户看到的页面
```

## 八、系统工作流程

### 用户登录流程

```
1. 用户打开网页，输入用户名和密码
2. 前端把密码发给后端
3. 后端去数据库查：这个用户名存在吗？密码对吗？
4. 如果对了 → 生成一个"门票"(JWT)，返回给前端
5. 前端保存这个"门票"，以后每次请求都带上
6. 后端看到"门票"就知道："哦，你是谁，你能干什么"
```

### 申请加入社团流程

```
1. 普通用户浏览社团列表 → 找到感兴趣的社团
2. 点击"申请加入" → 填写申请理由
3. 前端把申请信息发给后端
4. 后端把申请记录存到数据库的"入团申请表"里
5. 社长打开系统 → 看到待审核的申请列表
6. 社长点击"同意"或"拒绝"
7. 后端更新数据库里的申请状态
8. 申请人收到通知："您的申请已通过/已拒绝"
```

### 活动举办流程

```
1. 社长创建活动 → 填写标题、时间、地点、人数限制

3. 管理员审核 → 通过后状态变为"已发布"
4. 普通用户看到活动 → 点击"报名"
```

## 十一、系统的特色和创新点

### 1. 三种角色，各司其职

不是所有人都能做所有事。就像学校里：
- 学生只能看和申请
- 社长只能管自己的社团
- 管理员什么都管

这叫 **"权限控制"**，保证了系统的安全性。

### 2. 全流程审批机制

社团成立要审批、活动发布要审批、入团申请要审批。

就像现实中的办事流程：
> "先申请 → 再审核 → 最后生效"

### 3. 数据可视化

不是只有干巴巴的数字，而是用 **图表** 来展示：
- 饼图：各类型社团占比
- 柱状图：各社团人数对比
- 折线图：活动报名趋势

### 4. 前后端分离

前端和后端是 **两个独立的程序**，通过网络API对话。

好处：
- 前端后端可以分开开发，互不干扰
- 以后想做手机App，后端不用改
- 维护升级更方便
